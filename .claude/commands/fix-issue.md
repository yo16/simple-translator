# 問題修正パイプライン (TDDフロー)

ユーザーから指摘された問題: $ARGUMENTS

以下の手順を**順番通りに**実行する。各ステップは省略禁止。

> **⚠️ 絶対厳守**
> - 「簡単な修正」「明らかな修正」であってもテスト作成(ステップ3)を省略しない
> - 必ず「修正前にNG、修正後にOK」となるテストを作る（TDDのRed→Green）
> - 手動確認への委任は禁止

## ステップ1: Beadsタスク追加
Agent(`beads-manager`) を呼び出し、以下を含むタスクを作成:
- タイトル: 問題の要約
- 本文: 問題の詳細、再現手順、期待動作、修正方針
- ステータス: open
作成したタスクIDを記録する。

## ステップ2: ブランチ作成
Agent(`git-manager`) を呼び出し、devから `feature/bd-{id}` ブランチを作成してチェックアウトする。

## ステップ3: 失敗するテストの作成 【省略禁止】
問題領域に応じたテストエンジニアを呼び出す:
- フロントエンド → Agent(`frontend-test-engineer`)
- バックエンド → Agent(`backend-test-engineer`)

指示内容:
- 修正前の**現状のコードでは失敗する**テストを作成する
- 修正後の**期待動作を満たすと成功する**テストにする
- 正常系・異常系・境界値を適切にカバー
- テストの意図をコメントで明記

## ステップ4: テスト失敗の確認 (Red) 【省略禁止】
同じテストエンジニアに `npm test` (または該当コマンド) を実行させ、ステップ3で作成したテストが**実際に失敗すること**を確認する。

- 失敗しない場合 → テストが問題を捉えていない。ステップ3に戻る
- 想定通り失敗 → ステップ5へ

## ステップ5: 修正実装
問題領域に応じた実装エンジニアを呼び出す:
- フロントエンド → Agent(`frontend-engineer`)
- バックエンド → Agent(`backend-engineer`)
- スタイリング → Agent(`web-designer`)
- その他 CLAUDE.md の技術スタックに従う

ステップ3で作成したテストをパスさせることを明確な完了条件として指示する。

## ステップ6: テスト成功の確認 (Green) 【省略禁止】
テストエンジニアにテストを再実行させ、ステップ3のテストが**成功する**ことを確認する。併せて既存テストもすべて通ることを確認（リグレッションなし）。

- 失敗 → ステップ5に戻り修正を継続（3回失敗したら `/dev-rollback {id}`）
- 成功 → ステップ7へ

## ステップ7: 仕様書の更新 【省略禁止】
今回の修正で対応した内容は、ユーザーが要件として期待していたが元の仕様書には記載されていなかった機能・挙動である。**仕様書漏れ**として扱い、該当ドキュメントに追記する。

手順:
1. `docs/requirements.md` および `docs/design/` 配下のドキュメントを確認し、今回の修正内容がどこに属するかを判断する
   - ユーザ要求レベルの機能追加 → `docs/requirements.md`
   - 設計レベル（レイアウト、コンポーネント、データモデル、メッセージ仕様、エッジケース等）→ `docs/design/` の該当ドキュメント
   - 両方に関わる場合は両方を更新する
2. 該当箇所に今回の機能・挙動・制約を追記する。Beadsタスク ID を参照コメントとして記録する（例: `（bd-xxx で追加）`）
3. 設計ドキュメントに新たな関数・定数・状態フィールドを追加した場合は、その説明も記載する

**注意:** 設計の大幅変更が必要な場合はユーザに確認する。軽微な追記であればそのまま更新してよい。

## ステップ8: devへマージ
1. Agent(`beads-manager`): タスクをクローズ
2. Agent(`git-manager`): featureブランチにコミットし、devブランチへマージ（ステップ7の仕様書更新も同じコミットに含める）
